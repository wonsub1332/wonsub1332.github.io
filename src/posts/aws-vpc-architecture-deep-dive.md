# 실무 수준의 AWS VPC 설계: 고가용성과 보안의 정석

클라우드 인프라 구축의 첫 단추는 언제나 네트워크입니다. 잘못 설계된 네트워크는 추후 확장의 걸림돌이 되거나 치명적인 보안 허점을 야기합니다. 

이 포스트에서는 제공된 다이어그램을 바탕으로, **실무에서 가장 선호되는 VPC 아키텍처**를 왜 이렇게 설계하는지, 그리고 각 구성 요소가 어떻게 유기적으로 작동하는지 심층적으로 분석합니다.

---

## 1. 아키텍처 조감도 분석

![AWS VPC Architecture](/path/to/your/image.png)

이 아키텍처의 핵심 목표는 **'보안(Security)'**과 **'가용성(Availability)'**입니다. 

*   **리전**: 서울(`ap-northeast-2`)
*   **가용 영역(AZ)**: 2a, 2c (물리적으로 격리된 데이터 센터 활용)
*   **네트워크 분리**: 서비스용(Public)과 데이터/내부로직용(Private)의 엄격한 분리

---

## 2. IP 주소 설계 (CIDR Planning)

네트워크 설계의 시작은 IP 대역 할당입니다.

### VPC CIDR: `10.0.0.0/16`
*   총 **65,536개**의 IP 주소를 확보합니다. 
*   기업의 전체 네트워크 규모를 고려하여 다른 VPC나 사내 망과 겹치지 않도록 설정하는 것이 중요합니다.

### 서브넷 설계 (Subnetting)
각 서브넷을 `/24`로 쪼개어 관리 효율성을 극대화합니다. (각 서브넷당 251개 사용 가능 - AWS 예약 주소 5개 제외)

| 서브넷 명칭 | CIDR | AZ | 특징 |
| :--- | :--- | :--- | :--- |
| **Public Subnet 1** | `10.0.0.0/24` | 2a | NAT Gateway 및 로드밸런서 배치 |
| **Public Subnet 2** | `10.0.1.0/24` | 2c | 외부 노출 웹 서버 배치 |
| **Private Subnet 1** | `10.0.2.0/24` | 2a | 내부 애플리케이션, 프라이빗 웹 서버 |
| **Private Subnet 2** | `10.0.3.0/24` | 2c | 데이터베이스(RDS), 내부 캐시(ElastiCache) |

---

## 3. 외부 연결의 핵심: IGW vs NAT Gateway

많은 입문자가 혼동하는 두 게이트웨이의 차이점을 명확히 짚고 넘어갑니다.

### 인터넷 게이트웨이 (IGW)
*   **특징**: 양방향(Inbound/Outbound) 통신을 지원합니다.
*   **역할**: VPC와 공용 인터넷 사이의 문 역할을 합니다. 퍼블릭 서브넷에 위치한 리소스(EIP 또는 퍼블릭 IP 보유)가 인터넷과 직접 통신하게 해줍니다.

### NAT 게이트웨이 (NAT GW)
*   **특징**: 단방향(Outbound Only) 통신을 지원합니다.
*   **역할**: 프라이빗 서브넷의 서버가 외부로 패킷을 보낼 수는 있지만, **외부에서 프라이빗 서버로 직접 들어오는 것은 원천 차단**합니다.
*   **배치**: 반드시 인터넷과 연결된 **퍼블릭 서브넷**에 배치되어야 합니다.

---

## 4. 라우팅 테이블(Route Table) 상세 로직

데이터 패킷이 이동하는 경로를 정의합니다. 이 아키텍처에서는 두 종류의 라우팅 테이블이 필요합니다.

### ① 퍼블릭 라우팅 테이블 (Public Route Table)
| Destination | Target | 설명 |
| :--- | :--- | :--- |
| `10.0.0.0/16` | local | VPC 내부 통신 (기본값) |
| **`0.0.0.0/0`** | **igw-xxxxxx** | 그 외 모든 인터넷 트래픽은 **IGW**로 전송 |

### ② 프라이빗 라우팅 테이블 (Private Route Table)
| Destination | Target | 설명 |
| :--- | :--- | :--- |
| `10.0.0.0/16` | local | VPC 내부 통신 (기본값) |
| **`0.0.0.0/0`** | **nat-xxxxxx** | 인터넷 요청 시 **NAT 게이트웨이**를 경유 |

## 5. 계층적 보안 설계 (Security Groups & NACL)

보안은 한 겹이 아니라 여러 겹으로 쌓아야 합니다.

### 보안 그룹 (Security Group) - 인스턴스 단위
*   **Web Server (Public)**: 포트 80/443 허용 (Source: `0.0.0.0/0`)
*   **Private App (Private)**: 포트 8080 허용 (Source: **Web Server의 보안 그룹 ID**)
    *   *Tip: IP 주소가 아닌 보안 그룹 ID를 소스로 지정하면, 웹 서버가 증설되어 IP가 바뀌어도 보안 정책을 수정할 필요가 없습니다.*

### 네트워크 ACL (NACL) - 서브넷 단위
*   보안 그룹이 '허용' 중심이라면, NACL은 특정 IP 대역을 '차단'하거나 서브넷 전체의 경계를 지키는 용도로 사용합니다. (Stateless 방식)

---

## 6. 고가용성(HA)을 위한 추가 고려 사항

현재 다이어그램에서 한 단계 더 나아간다면 다음과 같은 요소를 고려해야 합니다.

1.  **Multi-NAT Gateway**: 현재 NAT GW가 한쪽 AZ(2a)에만 있습니다. 만약 AZ 2a 전체에 장애가 나면 AZ 2c의 프라이빗 서브넷도 인터넷 통신이 불가능해집니다. 실무에서는 각 AZ마다 NAT GW를 두어 완벽한 격리를 실현합니다.
2.  **ALB (Application Load Balancer)**: 사용자 요청을 Public Subnet 1, 2에 분산 배치된 웹 서버로 적절히 나누어주는 역할을 추가해야 합니다.
3.  **RDS Multi-AZ**: 데이터베이스를 Private Subnet 1, 2에 걸쳐 주/부(Primary/Standby) 구조로 배치하여 데이터 안정성을 확보합니다.

---

## 마치며

이 아키텍처는 AWS가 권장하는 **'Cloud-Native'** 네트워크의 시작점입니다. 

퍼블릭 서브넷은 외부와의 접점으로서 최소한의 리소스만 배치하고, 실제 중요한 데이터와 로직은 프라이빗 서브넷이라는 안전한 성벽 안에 두는 것이 핵심입니다. 이 구조를 직접 Terraform이나 CloudFormation으로 코드로 관리(IaC)해본다면 더욱 깊은 이해를 얻으실 수 있을 것입니다.
