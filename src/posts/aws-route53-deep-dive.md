# AWS Route 53: 클라우드의 똑똑한 내비게이션

도메인을 구매하고 IP에 연결하는 단순한 DNS 서비스를 넘어, **Route 53**은 AWS 에코시스템에서 트래픽을 지능적으로 제어하는 중추적인 역할을 합니다. 서비스의 가용성을 극대화하고 전 세계 사용자에게 최적의 응답 속도를 제공하기 위해 Route 53을 어떻게 활용해야 하는지 깊이 있게 살펴보겠습니다.

---

## 1. Route 53의 핵심 개념: Hosted Zone

Route 53을 시작하면 가장 먼저 만나는 개념이 **호스팅 영역(Hosted Zone)**입니다. 이는 도메인(`example.com`)과 그 하위 도메인(`www.example.com`, `api.example.com`)에 대한 DNS 레코드를 관리하는 컨테이너입니다.

*   **Public Hosted Zone**: 인터넷을 통해 접근 가능한 공개 도메인을 관리합니다.
*   **Private Hosted Zone**: 특정 VPC 내부에서만 유효한 도메인을 관리합니다. (예: 내부 마이크로서비스 간의 통신용 도메인)

### 주요 DNS 레코드 유형
기본적으로 알아야 할 레코드 타입들입니다:
- **A (Address)**: 도메인 이름을 IPv4 주소로 연결합니다.
- **AAAA**: 도메인 이름을 IPv6 주소로 연결합니다.
- **CNAME (Canonical Name)**: 도메인 이름을 다른 도메인 이름으로 연결합니다.
- **MX (Mail Exchange)**: 메일 서버를 지정합니다.
- **TXT (Text)**: 텍스트 정보를 포함하며, 주로 도메인 소유권 인증이나 보안 설정(SPF, DKIM)에 쓰입니다.

---

## 2. Alias(별칭) 레코드: AWS만의 특별함

일반적인 DNS에서는 `CNAME` 레코드를 사용하여 도메인을 다른 도메인으로 연결합니다. 하지만 Route 53에는 **Alias(별칭)** 레코드라는 강력한 기능이 있습니다.

### 왜 Alias 레코드를 써야 할까요?
1.  **Zone Apex 지원**: `CNAME`은 `example.com` 같은 루트 도메인(Zone Apex)에 직접 사용할 수 없지만, Alias는 가능합니다.
2.  **비용 절감**: Route 53 Alias 레코드를 통해 AWS 리소스(ELB, S3, CloudFront 등)로 향하는 쿼리는 **무료**입니다.
3.  **자동 업데이트**: 연결된 AWS 리소스의 IP 주소가 변경되어도 별도의 수정 없이 실시간으로 반영됩니다.

---

## 3. 지능형 라우팅 정책 (Routing Policies)

Route 53의 진정한 강력함은 상황에 맞는 트래픽 분산 능력에 있습니다.

### ① 단순 라우팅 (Simple Routing)
*   가장 기본적인 방식입니다. 하나의 도메인 이름에 대해 하나 이상의 IP 주소를 지정합니다. (여러 개 지정 시 랜덤하게 반환)

### ② 가중치 기반 라우팅 (Weighted Routing)
*   트래픽을 특정 비율로 나눕니다. (예: A 서버 80%, B 서버 20%)
*   **사용 사례**: 새로운 버전의 애플리케이션을 소수 사용자에게 먼저 테스트하는 **카나리(Canary) 배포**에 유용합니다.

### ③ 지연 시간 기반 라우팅 (Latency Routing)
*   사용자의 위치에서 네트워크 지연 시간이 가장 짧은 AWS 리전으로 트래픽을 보냅니다.
*   **사용 사례**: 글로벌 서비스를 운영할 때 전 세계 사용자에게 최상의 응답 속도를 보장합니다.

### ④ 장애 조치 라우팅 (Failover Routing)
*   **Active-Passive** 구조입니다. 주 서버에 장애가 발생하면 즉시 예비 서버로 트래픽을 전환합니다.
*   **사용 사례**: 재해 복구(DR) 시스템 구축의 핵심입니다.

### ⑤ 지리적 위치/근접성 라우팅 (Geolocation/Geoproximity Routing)
*   사용자가 접속한 국가나 대륙에 따라 다른 서버로 연결합니다.
*   **사용 사례**: 국가별 언어 설정이나 법규 준수(Data Residency)가 필요한 경우에 적합합니다.

### ⑥ 다중값 응답 라우팅 (Multivalue Answer Routing)
*   단순 라우팅과 비슷하지만, **상태 확인(Health Check)**을 통과한 레코드만 최대 8개까지 반환합니다. 
*   **특징**: 클라이언트 측 로드 밸런싱을 가능하게 하며, 가용성을 높이는 데 저렴하고 효과적인 대안입니다.

## 4. 상태 확인 (Health Checks) 및 장애 복구

Route 53은 단순히 주소만 알려주는 것이 아니라, 목적지가 건강한지 끊임없이 감시합니다.

### 상세 설정 및 원리
- **모니터링 대상**: 엔드포인트(IP/도메인), 다른 상태 확인 결과(계층적 상태 확인), CloudWatch 알람 등을 감시할 수 있습니다.
- **간격(Interval)**: 기본 30초(빠른 확인 시 10초)마다 체크합니다.
- **장애 임계값(Failure Threshold)**: 연속으로 몇 번 실패해야 'Unhealthy'로 판단할지 정합니다. (기본 3회)
- **문자열 매치**: 단순히 응답 코드뿐만 아니라, 특정 본문 텍스트가 포함되어 있는지까지 확인 가능합니다.

상태 확인 결과 '장애'로 판단되면, Route 53은 자동으로 해당 레코드를 DNS 응답에서 제외하고 정상적인 서버로 트래픽을 유도합니다.

---

## 5. 보안 및 관리 기능

### DNSSEC (Domain Name System Security Extensions)
DNS 쿼리에 대한 응답이 변조되지 않았음을 보증하기 위해 디지털 서명을 추가합니다. DNS 캐시 포이즈닝 공격으로부터 도메인을 보호하는 강력한 수단입니다.

### 쿼리 로깅 및 모니터링
CloudWatch Logs와 연동하여 누가, 언제, 어떤 도메인을 조회했는지 상세히 기록할 수 있습니다. 이는 보안 감사나 트래픽 패턴 분석에 매우 유용합니다.

---

## 6. VPC 아키텍처와의 결합 (Private DNS)

지난 포스트에서 다룬 [VPC 아키텍처](./aws-vpc-architecture-deep-dive.md)에서 Route 53은 내부 보안과 편의성을 동시에 잡는 도구가 됩니다.

*   **내부 서비스 명명**: `db.internal.local` 같은 도메인을 Private Hosted Zone에 등록하여, 복잡한 프라이빗 IP 대신 의미 있는 이름으로 서비스 간 통신을 수행할 수 있습니다.
*   **VPC Endpoint 연결**: AWS 서비스들을 인터페이스 엔드포인트로 연결할 때, Route 53을 통해 프라이빗 DNS 이름을 활성화하면 기존 코드 수정 없이 안전한 내부망 통신이 가능해집니다.

---

## 마치며: Route 53 선택이 아닌 필수인 이유

Route 53은 단순한 주소록이 아닙니다. 비즈니스의 **연속성(Health Check)**, **성능(Latency)**, 그리고 **비용 효율성(Alias)**을 결정짓는 핵심 인프라 서비스입니다. 

특히 대규모 트래픽을 다루거나 글로벌 서비스를 꿈꾼다면, Route 53의 고급 라우팅 정책과 보안 기능(DNSSEC)을 적재적소에 활용하는 능력이 필수적입니다. 지금 운영 중인 서비스의 도메인 설정을 다시 한번 점검해 보세요. 단순 라우팅만 쓰고 있다면, 성능 개선과 안정성 확보의 기회가 숨어있을지도 모릅니다.
