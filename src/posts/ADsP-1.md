## Chapter 01 데이터의 이해

### 001 데이터의 정의 및 유형

1. 데이터의 정의
    1. 추론과 추정의 근거를 이루는 사실
    2. 단순한 객체로서의 가치 뿐만 아니라 상호관계 속에서 갖는 것
    3. 객관적 사실 → 존재적 특성
    4. 추론, 예측, 전망, 추정을 위한 근거 → 당위적 특성
2. 데이터의 유형 - 예시와 특징 구분
    1. 정성적 데이터 (Qualitative Data)
        1. 언어, 문자
        2. 비정형 데이터
        3. 예) 설문조사의 주관식응답
    2. 정량적 데이터 (Quantitative Data)
        1. 수치, 도형, 기호
        2. 정형화된 데이터
        3. 예) 지역별 온도

### 002 암묵지와 형식지

1. 암묵지와 형식지 특징
    1. 암묵지
        1. 학습과 체험을 통해 개인에게 습득
        2. 공유되기 어려움
    2. 형식지
        1. 구체적이거나 문서화된 지식
        2. 공식적이고 체계적인 언어로 전달 가능 → 공유가 용이
2. 상호작용
    1. 지식 경영
        
        개인의 암묵지와 집단의 형식지가 상호작용하며 ‘생성 → 발전 → 전환‘ 되는 지식의 발전을 기반으로 하는 경영
        
    2. 암묵지와 형식지 전환 ( 공통화 → 표출화 → 연결화 → 내면화 :: `ㄱ 과 ㄴ 사이에 표연이 있다` )  
        
        → 순서 암기
        
        ![image.png](/assets/ADsP/part1/image.png)
        
        1. 공통화 : 암묵적 지식을 다른 사람에게 알려주는 것
        2. 표출화 : 암묵적 지식을 책이나 교본으로 만드는 것
        3. 연결화 : 책이나 교본에 자신이 알고 있는 새로운 지식을 추가하는 것
        4. 내면화 : 만들어진 책이나 교본을 보고 다른 직원들이 암묵적 지식을 습득하는 것

### 003 DIKW 피라미드

1. DIKW 계층구조
    1. 데이터, 정보, 지식을 통해 최종적으로 지혜를 얻어내는 과정
    2. 데이터 → 정보, 지식, 지혜
2. DIKW 의미           `⭐️각각에 대한 설명을 연결 할 수 있어야 함`

    3. 지식 (Knowledge) : 상호 연결된 정보 패턴을 이해하고 이를 토대로 예측한 결과물
    4. 지혜 (Wisdom) : 근본 원리에 대한 깊은 이해를 바탕으로 도출되는 아이디어

### 005 데이터베이스 용어 Basic

1. 데이터베이스 관련 용어
    1. Table : 단일 주제에 대해 행(row,record)과 열(column,field)로 구성되는 정보의 집합
    2. 관계형 데이터베이스 : 행(튜플,Tuple) 열(속성,Attribute) 
    3. 도메인 : 하나의 속성이 취할 수 있는 같은 타입의 원자 값들의 집합
    4. 카디날리티 : 릴레이션에서 튜플의 개수를 의미
    
    <aside>
    💡
    
    ![image.png](/assets/ADsP/part1/image%204.png)

        - ’학번~학점‘은 스키마이다
        - 튜플은 4개

2. DD, ERD
    1. DD (Data Dictionary)
        1. 자료에 관한 정보를 모아 두는 저장소
        2. 자료의 이름, 표현방식, 자료의 의미와 사용 방식, 다른 자료와의 관계를 저장
    2. ERD (Entity Relationship Diagram)
        1. 실체와 이들의 관계를 도형으로 표현
        2. 데이터베이스의 테이블이 Entity로 정의, Entity는 속성으로 구성
            
            → Relationship은 Entity 간의 관계를 의미, 두 엔티티 간에 선으로 표시
            

### 006 데이터베이스 용어 Advanced

1. Entity 간의 대응 관계의 종류
    
    ![image.png](/assets/ADsP/part1/image%205.png)
    
2. SQL (Structured Query Language)
    1. SQL의 개념
        - 데이터베이스를 구축하고 활용하기 위한 언어
        - RDBMS의 데이터를 관리하기 위해 설계된 특수 목적 프로그래밍 언어
            - D. 챔벌린과 레이먼드 F.보이스가 처음 개발
    2. SQL 명령어
        1. DDL (Data Definition Language)
            - 데이터 정의 언어 (테이블 구조 생성, 수정, 제거)
            - CREATE, ALTER, DROP, RENAME, TRUNCATE
        2. DML (Data Manipulation Language)
            - 데이터 처리 언어 (데이터 검색, 삽입, 수정, 삭제)
            - SELECT, INSERT, UPDATE, DELETE
        3. DCL (Data Control Language)
            - 데이터 제어 언어 (데이터베이스에 대해 접근 관리)
            - GRANT, REVOKE
        4. TCL (Tool Command Language)
            - 도구 명령 언어 (DML로 실행한 변경사항을 저장 관리)
            - COMMIT, ROLLBACK, SAVEPOINT
    3. SQL 문법
        1. SQL 집계 함수
            1. COUNT(*) : 데이터 타입에 상관없이 개수를 집계
            2. SUM(열), MIN(열), MAX(열), AVG(열) → 숫자 데이터에 대해 합계, 최소, 최대, 평균
        2. DML 중 SELECT 문
        
        ```sql
        SELECT 컬럼 FROM 테이블명 WHERE 조건식 [GROUP BY 그룹화 할 열] [HAVING 조건식] [ORDER BY 정렬 열]
        ```
        
3. 메타데이터, 인덱스
    1. 메타데이터(Metadata) `⭐️빈번하게 출제`
        1. 데이터에 관한 데이터
            
            데이터의 특성, 구조, 포맷, 원천, 의미 등을 설명
            
        2. 데이터를 분석하고 관리하는 데 도움이 되는 유용한 정보를 포함
    2. 인덱스(Index)
        1. 데이터베이스에서 테이븛의 동작 속도를 높여주는 자료구조
        2. 테이블 내의 1개 혹은 여러 개의 컬럼을 이용해 생성할 수 있다

### 007 DBMS (Database Management System)

1. DBMS의 개념
    1. 사용자와 데이터베이스 사이에서 사용자의 요구에 따라 정보를 처리해주고 데이터베이스를 관리
    2. 종류
        1. RDBMS ← 관계형 (제일 많이 사용)
        2. ODBMS ← 객체 지향, 복잡하다

2. DBMS의 장단점
    1. 장점
        1. 일관성과 무결성 유지를 통해 중복 최소화, 데이터 신뢰성을 높임
        2. 모든 사용자의 데이터 공유와 동시 접근이 가능
        3. 동시에 트랜잭션하는 경우에도 즉각적으로 결과값을 확인
        4. 액세스 관한을 통해 보안 제공
        5. SQL같은 언어를 통해 쉽게 접근 및 조작
    2. 단점
        1. 데이터베이스 관리로 모든 데이터 문제를 해결X
        2. 도입 및 유지에 비용 발생
        3. 구축 및 유지 보수, 관리에 전문지식 필요
        4. 시스템 문제 발생시 데이터 위험
    
    <aside>
    💡
    
    트랜잭션 (Transaction)?

    </aside>
    
3. NoSQL(Non-SQL, Non-Relational, Not Only SQL)
    1. 관계형 데이터베이스보다 덜 제한적인 일관성 모델
    2. 디자인 단순화, 수평적 확장성, 세세한 통제 등을 포함
    3. 종류 ) MongoDB, Apache HBase, Redis, Apache Cassandra

### 008 데이터베이스 설계

1. 데이터베이스 설계 절차 **`⭐️요 개 논 물⭐️`**
    
    ```mermaid
    flowchart LR
    node(요구 조건 분석)
    node2(개념적 설계)
    node3(논리적 설계)
    node4(물리적 설계)
    node-->node2-->node3-->node4
    
    ```
    
    - 요구 조건 분석
        - 데이터베이스 사용자, 사용 목적, 사용 범위, 제약 조건 등을 정리하여 명세서를 작성
    - 개념적 설계
        - 정보를 추상적인 개념을 표현하는 과정
        - DBMS에 독립적인 개념적 데이터 모델을 생성함
        - 결과물 : E-R 다이어그램 (Entity Relationship Diagram)
    - 논리적 개념
        - 데이터 모델을 구체적인 스키마로 변환하여 데이터 구조를 정확하게 정의하거, 데이터 모델을 정규화하여 테이블을 관리 가능한 크기와 논리적 관계로 나눔
        - 자료를 컴퓨터가 이해할 수 있도록 특정 DBMS의 논리적 구조로 변환
    - 물리적 설계
        - 데이터를 저장할 장치를 선택하고 인덱스를 구성하여 데이터 검색을 최적화함
        - 데이터베이스의 보안, 백업 및 복구 전략을 개발하여 데이터의 안전성 보장함

### 009 시대별 기업 내부 데이터베이스 솔루션 ⭐️⭐️⭐️

1. 1980년대 기업 내부 데이터베이스 솔루션 `두가지 용어 구분`
    1. OLTP (On-Line Transaction Processing, 온라인 거래 처리)
        
        주 컴퓨터와 통신 회선으로 접속된 복수의 사용자 단말에서 발생한 트랜잭션을 주 컴퓨터에서 처리
        
        그 결과를 사용자에게 되돌려 보내주는 처리 형태
        
        예) 상품 주문, 회원 정보 수정, 은행 거래
        
    2. OLAP (On-Line Analytical Processing, 온라인 분석 처리)
        1. 다차원으로 이루어진 데이터로부터 통계적인 요약정보를 제공할 수 있는 기술
        2. 다차원의 데이터를 대화식 질의를 통해 분석하는 정보 분석용 소프트웨어
            
            예) 10년간 A사의 직급별 임금 상승률 등
            
        
        `OLTP는 실시간 트랜잭션 데이터의 처리 / OLAP는 대량의 데이터에 대한 복잡한 분석과 보고 지원`
        
2. 2000년대 기업 내부 데이터베이스 솔루션
    1. CRM (Customer Relationship Management)
        1. 고객별 구매 이력 데이터베이스를 분석하여 고객에 대한 이해를 돕고,
            
            이를 바탕으로 각종 마케팅 전략을 실행하여 높은 이익을 창출
            
        2. 고객 데이터를 중심으로 구축, 고객에 대한 기록, 요구 사항, 구매 이력, 서비스 요청 및 마케팅 활동 등을 추적,분석하여 개인화된 고객 경험 제공
    2. SCM (Supply Chain Management)
        1. 제조, 물류, 유통 업체 등 유통 공급망에 참여하는 모든 업체들의 협력을 바탕으로 정보기술을 활용하여 재고를 최적화하기 위한 솔루션
        2. 기업이 외부 공급 업체 또는 제휴 업체와 통합된 정보 시스템으로 연계하여 시간과 비용을 최적화하기 위한 것이다
    3. SCM = 물류 공급 / CRM = 고객 에 대한 관리를 목표

### 010 분야별 기업 내부 데이터베이스 솔루션

1. 제조부문 기업 내부 데이터베이스 솔루션
    1. 데이터웨어하우스 (Data Warehouse) ↔ `데이터 마트와 구분`
        1. 기업 내의 의사결정 지원 애플리케이션을 위한 정보를 제공하는 하나의 통합된 데이터 저장 공간
        2. 시간적 흐름에 따라 변화하는 값이며 일정 기간 유지
        3. 4대 특징⭐️⭐️
            1. 데이터의 통합 : 데이터들은 전사적 차원에서 일관된 형식으로 정의
            2. 데이터의 시계열성 : 시간의 흐름에 따라 변화하는 값
            3. 데이터 주제 지향적 : 특정 주제에 따라 데이터들이 분류, 저장, 관리

### 011 빅데이터(Big Data)

1. 빅데이터의 정의
    1. 일반적인 데이터베이스 소프트웨어로 저장, 관리, 분석할 수 있는 범위를 초과하는 규모의 데이터

    3. 대용량 데이터를 활용해 작은용량에서는 얻을수 없었던 새로운 통찰이나 가치를 추출하는 일
    4. 나아가 이를 활용해 시장, 기업 및 시민과 정부의 관계 등 많은 분야에 변화를 가져오는 일

    2. 4V : 3V + Value, ROI(Return On Investment, 투자자본수익률) 관점에서 보는 빅데이터
    
    | Volume | - 데이터의 크기, 생성되는 모든 데이터를 수집
    - 구글 실시간 자동 변역 시스템에 적용되는 빅데이터의 특징 |
    | --- | --- |
    | Variety | - 데이터의 다양성을 의미
    - 정형화된 데이터를 넘어, 텍스트, 오디오, 비디오와 같은 비정형 데이터 및 웹 문서와 같은 반정형 데이터를 대상으로 함 |
    | Velocity | - 데이터의 속도를 의미함
    - 사용자가 원하는 시간 내에 데이텅 분석 결과를 제공하며, 업데이터 속도가 빠름 |
    | Value | - Value : 비즈니스 효과 요소
    - Volume, Variety, Velocity : 투자 비용 요소 |

### 016 빅데이터의 활용 기법

1. 활용 기법 1  ⭐️ `정의 뿐만 아니라 예시도 같이` 
    1. 연관 규칙 학습 (Association Rule Learning)
        1. 변수간 주목할 만한 상관관계가 있는지 찾아내는 기법
        2. 시스템 로그 데이터를 분석해 침입자나 유해 행위자 색출에 활용
        
        Ex) 우유구매자가 기저귀도 같이 구매하는가?
        
    2. 유형 분석 (Classification Tree Analysis)
        1. 범주를 찾아낸느 통계적 분류를 위해 기존 자료를 바탕으로 만들어진 훈련용 분류틀이 있어야 한다
        2. 문서를 분류하거나 조직을 그룹으로 누눌때, 온라인 수강생들을 특성에 따라 분류할 때 사용
        
        Ex) 이 사용자는 어떤 특성을 가진 집단에 속하는가?
        
    3. 유전 알고리즘 (Generic Algorithms)
        1. **최적화**가 필요한 문제의 해결책을 자연선택, 돌연변이 등과 같은 메커니즘을 통해 점진적으로 진화 시켜나가는 방법
        
        Ex) 시청률을 얻으려면 어떤 프로그램을 어떤 시간대에 방송해야 하는가?
        
2. 활용 기법 2 `감정분석과 소셜 네트워크 분석을 구분`
    1. 기계 학습 (Machine Learning)
        1. 훈련 데이터로부터 학습한 알려진 특성을 활용해 ‘예측’하는 일에 초점을 맞춘다
        2. 예) 스팸 메일, 넷플릭스 추천
    2. 회귀 분석 (Regression Analysis)
        1. 선형 함수로 나타낼 수 있는 수치 데이터 분석
        2. 예) 사용자의 만족도가 충성도에 어떤 영향을 미치는 가?
    3. 감정 분석 (Sentiment Analysis)
        1. 특정 주제에 대해 말하거나 글을 쓴 사람의 감정을 분석
        2. 예) SNS를 바탕으로 고객이 원하는 것을 찾아낼 때 활용
    4. 소셜 네트워크 분석
        1. 사회 관계망 분석이라고도 한다
        2. 영향력 있는 사람을 찾아낼 수 있으면, 사람들 간 소셜 관계를 파악할 수 있음
        3. 예) 보험사에서 보험사기 적발

### 017 빅데이터의 위기 요인과 통제 방안

1. 빅데이터의 위기 요인

위기 요인

- 사생활 침해
- 책임 원칙의 훼손
- 데이터 오용

통제 방안

- 동의제 → 책임제로 전환
- 기존 책임 원칙 강화
- 접근권 및 객관적 인증 방안 도입
1. 사생활 침해
    1. 위기 요인
        1. 우리 주위에 센서들의 수가 늘어나 정보가 수집되어 다른 목적으로 사용 가능
        2. 방지를 위해 익명화 기술이 발전중 → 아직 충분 X
            
            <aside>
            <img src="https://www.notion.so/icons/link_gray.svg" alt="https://www.notion.so/icons/link_gray.svg" width="40px" />
            
            익명화(Anonymization) vs 가명(Pseudonym)
            
            - 익명화 : 개인 식별 정보를 삭제하거나 알아볼 수 없는 형태로 변환하는 방법에 대한 포괄적 개념
            - 가명 : 다른 추가 정보가 있으면 특정 개인 유추가 가능, 개인정보 비식별화의 한가지 방법
            </aside>
            
    2. 통제 방안
        1. 동의제 → 책임제
        2. 개인정보 제공자의 동의보다 개인 정보 사용자에게 책임을 묻는다
2. 책임 원칙의 훼손
    1. 위기 요인
        1. 분석 대상이 되는 사람들은 예측 알고리즘의 희생량이 될 가능성이 증가한다
        2. 민주주의 사회 원칙 크게 훼손 가능
        3. 예) 범죄 예측 프로그램을 통해 범죄가 발생하기 전의 전 체포
    2. 통제 방안
        1. **기존의 책임 원칙을 강화**
        2. 결과 기반의 책임원칙 고수
3. 데이터의 오용
    1. 위기 요인
        1. 빅데이터는 일어난 일에 대한 데이터에 의존한다
        2. 항상 예측이 맞을 수는 없다
    2. 통제 방안
        1. 데이터 알고리즘에 대한 접근권을 허용해야 한다

### 018 개인정보 비식별화 기법

1. 개인정보 비식별화 기법
    1. 가명처리
        - 식별이 가능한 데이터 → 개인 식별이 불가능한 다른 값 대체
    2. 총계 처리
        - 개인정보레 통곗값을 적용하여 개인을 특정할 수 없게 하는 방법
            - 집계 수량이 적으면 개인정보 예측 가능 → 데이터 양이 많아야 함
        - 집계 처리되어 정밀한 분석이 어렵고, 재배열 방법의 경우 개개인의 특성을 파악하기 어려움
    3. 데이터 값 삭제
        - 특정 데이터 값의 부분 또는 전체를 삭제
        - 식별자 삭제, 식별자 부분 삭제, 레코드 삭제(=행 삭제), 식별 요소 전부 삭제 등의 방법이 있다
    4. 데이터 범주화
        - 식별 값을 해당 그룹의 대푯값이나 구간 값으로 변화하는 방법
            
            예 ) 홍길동 → 홍 씨 , 35세 → 30~40세
            
    5. 데이터 마스킹 (Data Masking)
        - 식별 값의 전체 또는 부분을 대체 값으로 변환하는 방법
        - 임의의 잡음을 추가, 공백으로 대체
            
            예) 카드 뒤 4자리 숨기기,  홍*동
            
    6. 잡음 첨가
        1. 자료의 값에 잡음을 추가하거나 곱하여 원래 자료에 약간의 변형을 가하여 공개하는 방법

    1. 모든 분석은 가정에 근거하므로 잘못된 분석을 할 수 있고, 잘못된 분석은 안 좋은 결과를 가져올 수 있으므로 인문학적인 사고가 필요
    2. 데이터 사이언스를 위한 인문학적인 사고
        1. 모델의 능력에 대해 항상 의구심
        2. 가정과 현실의 불일치에 대해 계속 고찰

    1. 신속한 의사결정을 원하는 비즈니스에서는 실시간 ’상관관계’ 분석에서 도출된 인사이트를 바탕으로 수익을 창출할 수 있는 기회가 점점 늘어나고 있음
    2. ‘상관관계’를 통해 특정 현상의 발생 가능성이 포착되고, 그에 상응하는 행동을 하도록 추천되는 일이 점점 늘어날 것
    3. 데이터 기반의 ‘상관관계’ 분석이 주는 인사이트가 ‘인과관계’에 의한 미래 예측을 점점 더 압도해 가는 시대가 도래
